YOUR TASK — STEP-BY-STEP INSTRUCTIONS
────────────────────────────────────────────────────────────────────────────────

STEP 1 — KEYWORD EXTRACTION
  Extract all ATS-critical keywords from the job description:
  - Job titles / role synonyms
  - Required hard skills (languages, tools, frameworks, platforms)
  - Preferred/bonus skills
  - Domain terms (e.g., "MLOps", "data pipeline", "NLP", "computer vision")
  - Soft skills explicitly mentioned (e.g., "collaboration", "communication")
  - Degree / certification requirements

STEP 2 — GAP ANALYSIS
  Compare extracted keywords against the candidate's background above.
  Identify:
  a) Exact matches — mirror the job's exact phrasing wherever truthful.
  b) Equivalent skills — map candidate's tools to job's preferred tools if
     functionally equivalent (e.g., "MLflow" ↔ "experiment tracking").
  c) Missing keywords — do NOT fabricate skills; note any genuine gaps.

STEP 3 — REWRITE THE PROFILE SUMMARY
  Write a 2–3 sentence profile summary that:
  - Opens with the exact job title from the posting.
  - Incorporates the top 5–7 ATS keywords naturally.
  - Quantifies impact where possible.
  - Avoids buzzwords with no substance.

STEP 4 — REWRITE TECHNICAL SKILLS
  - Keep ONLY skills that appear in or are strongly implied by the job description.
  - Remove any skill row that has zero relevance to the role.
  - Mirror the exact terminology used in the job posting (e.g., if the JD says
    "Apache Kafka", do not write "Kafka" — write "Apache Kafka").
  - Group skills logically and keep the table compact.

STEP 5 — REWRITE WORK EXPERIENCE BULLETS
  For each bullet point:
  - Lead with a strong action verb matching job description language.
  - Embed 1–3 keywords from the job description per bullet (naturally, not
    keyword-stuffed).
  - Include at least one quantified result per role (already present; preserve
    or improve them).
  - Remove bullets that are entirely irrelevant to the target role.
  - Do NOT invent new experience; only reframe existing experience.

STEP 6 — SELECT & REWRITE PROJECTS
  - Keep at most 2 projects. Drop a project only if it adds zero ATS value
    compared to the other.
  - Rewrite project bullets to surface keywords from the job description.
  - Adjust Tech Stack line to lead with the most relevant tools for this role.

STEP 7 — ONE-PAGE ENFORCEMENT (CRITICAL RULE)
  The final LaTeX output MUST fit within one A4 page. Strategies to apply:
  - Remove bullets that are least relevant to the job (minimum 1 bullet per role).
  - Shorten remaining bullets: aim for ≤ 1.5 lines each in 10pt font.
  - Trim certifications: keep only those directly relevant to the role.
  - Do NOT change margins, font size, or document class settings.
  - Do NOT remove the header, education, or profile summary sections.

STEP 8 — OUTPUT (CV)
  Return the COMPLETE, ready-to-compile src/cv.tex file with ALL changes applied.
  Preserve:
  - All LaTeX preamble, packages, and custom environments (jobshort, joblong).
  - All hyperlinks (GitHub, LinkedIn, email, phone, location, project links).
  - The "Last updated: \today" footer.
  - The existing document structure and section order.
  Do NOT include any explanation or commentary — output raw LaTeX only.

TONE & STYLE RULES:
  - Professional yet personable — not stiff or robotic.
  - Confident, not arrogant.
  - No clichés: do NOT use phrases like "I am a hard worker",
    "passionate team player", "go-getter", or "think outside the box".
  - Keep total length to 300–400 words (body only, excluding header/sign-off).
  - Plain text format only — no markdown, no LaTeX, no bullet points.
  - Do NOT fabricate company names, hiring manager names, or role details
    not present in the job description.

OUTPUT:
  Write the full cover letter in cl.tex and compile it to get cl.pdf.
  Do NOT include any explanation or commentary — output the letter only.
